Personalized Agent Bandits with Factorized Orchestration Actions

A lightweight Bayesian bandit layer that personalizes how an agent uses memory, tools, clarification, and answer style.

PAB-F learns user-specific residual orchestration preferences from partial feedback while enforcing hard safety and feasibility constraints.

Overview

Personalizing the orchestration layer

PAB-F personalizes the orchestration layer of an agent: when to use memory, when to call tools, when to ask or confirm, and how to shape the answer.

Modern personal agents repeatedly choose intermediate execution actions before producing a final natural-language response. PAB-F models those choices as a safe contextual bandit problem: each request creates a context, the agent filters unsafe or infeasible actions, selects one orchestration action, and observes partial feedback only for that selected action.

The action is factorized as a = (m, τ, s), where m is memory mode, τ is tool mode, and s is answer behavior or style. Factorization lets feedback from one memory-tool-style action update reusable components instead of treating every action triple as an unrelated arm.

E[y | c,a] = b(c,a) + φ(c,a)Tθ - λ cost(c,a)

b(c,a)

Default system behavior and rule-based orchestration utility.

φ(c,a)Tθ

User-specific residual preference learned online.

λ cost(c,a)

Explicit penalty for latency, tools, clarification, or verbosity.

Method

Safe factorized residual learning

PAB-F is designed as an orchestration plugin for a larger assistant runtime. The host agent still executes memory retrieval, tool calls, clarification, and final response generation.

1

Safe action layer

Filters actions by hard constraints such as memory permissions, available tools, risk level, and confirmation requirements. Exploration only happens inside the safe set.

2

Factorized Thompson sampling

Scores safe actions using default utility, sampled user residual preference, and explicit cost while sharing signal across memory, tool, and answer-style components.

3

Preference promotion

When posterior confidence and evidence are high, stable learned preferences can be written into memory or prompt policy as interpretable state.

Representative factorized action space

The current paper uses five memory modes, six tool modes, and six answer styles, producing 180 combinations before safety constraints.

Memory
no memory, recent memory, semantic memory, preference memory, profile summary
Tools
no tool, web search, file search, code execution, preference checker, ask user
Styles
direct, concise, detailed, step-by-step, ask clarification, confirm first

Algorithm

Online personalization loop

The loop residualizes feedback against default behavior and explicit cost, then updates a Bayesian posterior over user-specific orchestration preferences.

Limitations

Limitations and next steps

Numerical stability

Thompson sampling with multivariate Gaussian sampling can fail if posterior covariance becomes ill-conditioned, especially when hard constraints permanently block some feature directions. Future fixes include covariance regularization, diagonal jitter, Cholesky fallback, or low-rank/diagonal posterior options.

Preference promotion precision

With only 30 rounds per user in PAHF, few preferences may reach the promotion threshold. Future work should use calibrated thresholds, more evidence, hierarchical sharing, or confidence intervals tailored to sparse evidence.

Non-stationary coverage

Current runs cover stationary and single-drift scenarios. Future evaluations should include continuous persona evolution and stochastic preference drift.

Runtime overhead

Math500 adaptive methods are substantially slower than Direct and Direct-Pref in the current pilot. Caching, smaller candidate sets, cheaper judges, and async evaluation are natural next steps.
